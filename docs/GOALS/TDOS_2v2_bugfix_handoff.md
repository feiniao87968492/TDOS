## P0：必须修复的问题

### 1. 左侧按钮闪烁、无法稳定点击

#### 已确认的代码原因

`src/online.js` 的 `setBattleControlsEnabled()` 会遍历 `#battleControls` 内所有 `button/select/input`，统一覆盖它们的 `disabled`：

- `src/online.js:579-587`

每次收到快照，`handleSnapshot()` 都会调用 `applyViewerControlState()`：

- `src/online.js:1875-1907`

后者再次调用 `setBattleControlsEnabled(true)`。与此同时，约 10Hz 的 `updateBattleStatus()`、`syncShipSelectOptions()` 和 `updateSkillButtons()` 又会根据分离等级、能量、冷却、舰船生存状态重新禁用部分按钮。

因此存在两个状态写入者：

1. 网络快照层周期性把全部控件启用；
2. HUD/技能层周期性按规则重新禁用。

它们以不同频率反复覆盖 DOM，表现就是按钮亮暗闪烁、按下时状态变化、部分操作难以触发。AI 房也走相同的在线战斗 UI，因此同样受到影响。

#### 修复目标

- 把“整个面板是否允许操作”和“某个按钮当前是否合法”分离。
- 不允许全局开关在每个快照上无条件把所有子控件设为 `disabled = false`。
- 最终每个按钮的禁用态只能由一个清晰、可测试的状态计算过程决定。
- 控件状态没有变化时，不要重复写 DOM。
- 舰队被歼灭后可禁用战斗指令，但观察、缩放、返回等非战斗功能仍应可用。

具体实现可以是统一的 `renderControlState()`、状态 reducer、分组禁用或其他更合适的结构，不要求保留现有函数形式。

### 2. 同盟舰队航线被画到同一个目标点

#### 已确认的代码原因

本地航线预测缓存 `app.routeOverrides` 只用 `ship.key` 作键：

- `src/online.js:2051-2061`
- `src/online.js:2095-2157`
- `src/online.js:2347-2355`

四支舰队的舰船键都会重复使用 `main/sub1/sub2`。渲染 2v2 时，`src/battle/render.js:954-976` 会遍历所有友军舰队；而 `getDisplayRouteForShip(team, ship)` 最终按 `ship.key` 读取本地预测。

结果是：A1 为自己的 `main` 设定预测航线后，渲染 A2 的 `main` 时也会读取同一份 override，并以 A2 主舰为起点画向 A1 的目标点。服务端模拟状态本身未必被串改，但客户端画面已经发生跨舰队污染。

#### 修复目标

- 所有客户端临时状态必须用能够唯一标识舰船的键，例如 `fleetSeat + shipKey` 或稳定实体 ID。
- 如果本地预测只属于当前玩家，也可以在读取时明确限定 `team.seat === app.seat`。
- 清理、ACK、拖拽和超时逻辑必须采用同一套命名空间，不能只修渲染读取。
- 队友航线可以共享显示，但应与自己的可编辑航线有明显区别；队友航线不能出现可编辑手柄或造成“我能控制它”的错觉。
- 验证服务端权威状态：A1 的所有 action 都不得改变 A2、B1、B2 的舰船路线、油门、分离等级、能量、技能冷却或选舰状态。

### 3. 2v2 结算双方都显示失败

#### 已确认的代码原因

战斗内核在 2v2 结束时使用联盟作为胜者：

- `shared/game-core.js:6409-6422`
- `winnerAllianceId` 和 `winnerSeat` 的值为 `"A"` 或 `"B"`。

客户端却在 `src/online.js:1095-1125` 使用：

```js
winnerSeat === app.seat
```

玩家席位是 `"A1" / "A2" / "B1" / "B2"`，所以任何玩家都不可能与 `"A" / "B"` 相等，四个人都会进入失败分支。

#### 修复目标

- 2v2 以 `winnerAllianceId` 与 `app.allianceId` 判断胜负。
- 1v1 仍保持原有 A/B 席位判断。
- A1 被全歼、A2 最终获胜时，A1 和 A2 都显示联盟胜利；B1、B2 都显示失败。
- 平局、观战、无胜者数据和断线后收到延迟结算都要有明确行为。

#### 同一结算模块中的附带错误

`src/online.js:1145-1157` 仍只查找 `seat === "A"` 和 `seat === "B"`。2v2 房间只有 A1/A2/B1/B2，因此结算阵容、玩家名和 ID 会缺失或错误回退。

请把 2v2 结算真正改成“联盟对联盟”，展示每方两名玩家及其阵容；不要求沿用当前双卡片布局。

### 4. 左侧下半部分控件重叠

#### 高概率根因

当前侧栏采用固定视口高度并允许滚动：

- `styles.css:581-590`

但舰队区域使用可压缩 flex：

- `.battle-panel .fleet-section { flex: 1 1 auto; min-height: 0; }`
- `.fleet-roster { flex: 1 1 auto; min-height: 0; }`

三个 `.fleet-row` 又各自要求 `min-height: 50px`。当可用高度不足时，父级继续收缩，子级内容以可见 overflow 溢出，容易覆盖后面的队内通信区。即使 AI 房隐藏通信区，在较矮视口、较长本地化文本或控件内容增高时也可能溢出。

#### 修复目标

- 侧栏在目标分辨率下不允许任何按钮、舰队卡、通信控件互相覆盖。
- 高度不足时必须形成一个清晰的滚动区域，而不是让 flex 子项越界。
- 中文、英文、较长角色/技能名都要测试。
- 不要通过把字号压得难以阅读来“修复”。
- 可以重新划分滚动区域、取消舰队卡强制等高、做折叠区或调整桌面 HUD 结构。

## P1：审计中发现的其他高风险问题

### 5. 刷新页面后重连可能长时间无法控制

服务端在重连槽位中保留 `lastProcessedSeq`：

- `server/server.js:267-285`
- `server/server.js:798-815`

客户端重连票据只保存房间、席位和 token：

- `src/online.js:407-447`

新页面中的 `app.seq` 从 0 开始，而服务端会丢弃 `seq <= lastProcessedSeq` 的输入：

- `src/online.js:206-220`
- `server/server.js:941-988`

因此玩家刷新页面后成功回到原席位，并不代表新的战斗指令会立即生效。现有重连测试只在恢复后发送了 `select_ship`，没有发送带序号的战斗输入，所以没有覆盖该问题。

请设计可靠的序号续接方案，并增加真实输入回归测试。不要让玩家靠连续点击直到序号追上旧值。

### 6. 过滤快照仍携带所有敌方的选舰信息

服务端对 2v2 舰队和敌方实体做了按联盟视野过滤，但随后无条件加入整个房间的：

```js
selectedShips: selectedShipsForRoom(room)
```

见 `server/server.js:1002-1029`。

这会把 B1/B2 当前选中的舰船键发送给 A 阵营，即使敌舰完全不可见。当前 UI 未必展示，但协议层已经泄露不该有的对手操作信息。

请审查快照中所有玩家级元数据，遵循：

- 自己与队友的信息可共享；
- 敌方不可见舰队不应泄露选舰、路线、技能状态、资源、角色身份或坐标；
- 观战数据另行处理。

### 7. 战斗开始后服务端仍接受阵容和昵称修改

`server/server.js:1140-1165` 对 `set_name`、`set_loadout` 没有房间阶段校验。客户端虽然在倒计时/战斗时锁住阵容 UI，但自定义客户端仍能改变服务端玩家记录；模拟中的阵容不会同步变化，房间列表或最终结算却可能读取到更新后的记录，产生显示与真实对局不一致。

请决定并实现明确规则：通常倒计时开始后冻结昵称和阵容，直到离开或本局结束。

## P2：允许结合体验自主优化的项目

### 队友航线的视觉语义

当前会绘制所有友军航线。建议保留共享战术信息，但区分：

- 自己：高亮、可见控制点、可编辑；
- 队友：较淡或虚线、无控制点、不可编辑；
- 已被歼灭的观察者：可查看队友路线，但不能发战斗指令。

具体风格可以自行设计。

### 队内目标标记

点位通信直接取 `app.pointer` 作为落点。请验证：

- 点击侧栏按钮时是否使用了用户预期的位置；
- 移动端没有最近地图指针时是否会错误落在默认中心；
- 是否需要“选择通信类型 → 点击地图确认”的瞄准流程；
- 标记是否只发给同盟，是否有频率限制、过期与重复事件处理。

### 友军碰撞

`shared/game-core.js:6297-6299` 直接跳过同联盟舰船碰撞。它不会造成控制权串用，但允许两名队友的舰船完全重叠，可能带来辨识、集火和遮挡问题。请把它作为明确的玩法决定评估：保留穿透、做软分离，或做友军碰撞均可，但需要测试并记录理由。

## 已确认正常的基础边界

以下部分在当前实现和现有测试中表现正确，修改时不要破坏：

- 服务端从连接对应的玩家席位推导控制对象，不接受 action 内伪造的 `fleetSeat`。
- `applyActionForSeat("A1", action)` 会把 action 交给 A1 的 Team，而不是 A2。
- A1 单支舰队被全歼不会立即结束比赛；A 联盟所有舰队均失去存活舰船后才判 B 获胜。
- 同盟共享视野由两支友军舰队的传感器合并计算。
- 队内通信由服务端推导发送者和联盟，不广播给敌方。

不要因为客户端航线显示串线就把问题误判成服务端控制权已经完全串用；但仍需用更完整的自动化测试验证所有 action 类型。

## 验收标准

- AI 房和 2v2 左栏按钮不闪烁，合法操作一次点击即可触发。
- 任意目标分辨率下侧栏无重叠，内容不足时正常滚动。
- 两名队友可同时为同名舰船设置不同路线，显示与权威状态均不串线。
- 客户端无法控制队友舰队；伪造席位字段也无效。
- 结算按联盟判断，先被全歼的玩家仍继承最终联盟胜负。
- 2v2 结算正确展示四名玩家和四套阵容。
- 刷新后重连的第一条合法输入立即生效。
- 不可见敌方信息不会从快照旁路泄露。
- 现有 1v1、AI 房、共享视野、队内通信和移动端没有回归。
- 新增测试能够在恢复旧 Bug 时稳定失败。
